# Project Overview — Autonomous AI Job Search Agent

> **Phase 0 — Project Discovery**
> Status: Draft v0.1 · Owner: Nikhil Meshram · Last updated: 2026-05-30

---

## 1. Vision

An always-on, personal AI engineer that runs Nikhil Meshram's job search end-to-end —
discovering relevant roles, tailoring his application materials, applying with his
explicit approval, managing recruiter conversations, scheduling interviews, and getting
measurably better at all of it over time — so that a high-quality, full-time job search
runs in the background of a full-time job.

## 2. Mission

Replace the repetitive, high-volume, low-leverage parts of a job search (searching,
filtering, resume tailoring, cover-letter writing, form-filling, follow-up tracking)
with a supervised autonomous agent, while keeping the human (Nikhil) in control of every
outward-facing action and every decision that affects his reputation.

## 4. Success Metrics

These are the north-star and supporting metrics the system optimizes and reports on.

### North Star
- **Qualified interviews per week** generated by the agent.

### Funnel metrics
| Metric | Definition | Initial target |
|--------|-----------|----------------|
| Discovery volume | New unique relevant jobs surfaced / day | ≥ 30/day |
| Match precision | % of surfaced jobs Nikhil marks "relevant" | ≥ 70% |
| Application throughput | Approved applications submitted / week | ≥ 25/week |
| Response rate | Recruiter responses ÷ applications | ≥ 8% |
| Interview rate | Interviews ÷ applications | ≥ 3% |
| Reply latency | Time from recruiter email → drafted reply | ≤ 15 min |
| Time saved | Manual hours displaced / week (estimated) | ≥ 10 hrs/week |

### Quality / safety metrics
- 0 applications submitted without an approval record (in HITL mode).
- 0 messages sent to the wrong recipient/thread.
- Resume-fabrication rate = 0 (no invented experience — see AI guardrails).

## 5. User Personas

### Persona A — "The Operator" (Primary: Nikhil)
- Senior full-stack engineer, time-poor, employed, doing a passive→active search.
- Wants leverage, not loss of control. Reviews drafts on mobile in spare minutes.
- Technical enough to self-host, read logs, and trust but verify.
- **Needs:** high signal-to-noise matches, near-zero embarrassing mistakes, one-tap approvals.

### Persona B — "The Recruiter / Hiring Manager" (External)
- Receives applications and emails generated via the system.
- Does **not** know (and need not know) automation is involved beyond what's honest.
- **Needs (that we must respect):** genuine, accurate, non-spammy, human-quality communication.

### Persona C — "The Maintainer" (Nikhil, later)
- Same person, different hat — extends the system, adds job boards, tunes prompts.
- **Needs:** clean architecture, docs, ADRs, tests, observability.

## 6. Functional Requirements (summary)

Full detail in `docs/requirements/`. High level:

- **FR-1 Discovery:** Poll/scrape configured job sources; normalize into a canonical Job record.
- **FR-2 Matching:** Score each job vs. profile; produce ranked queue with explanations.
- **FR-3 Resume tailoring:** Generate ATS-friendly, role-specific resume variant from a master profile — without fabricating facts.
- **FR-4 Cover letter:** Generate tailored cover letter per application.
- **FR-5 Apply:** Fill/submit applications via browser automation or APIs, gated by approval.
- **FR-6 Tracking:** Persist full lifecycle state for every job/application.
- **FR-7 Inbox monitoring:** Watch Gmail for recruiter activity; classify and link to applications.
- **FR-8 Reply drafting:** Draft context-aware replies; send after approval.
- **FR-9 Scheduling:** Read calendar availability, propose slots, create events.
- **FR-10 Learning:** Capture outcomes; feed back into matching/writing.
- **FR-11 Dashboard:** Human control plane for review, approvals, analytics, config.
- **FR-12 Notifications:** Alert Nikhil when his input is required.

## 7. Non-Functional Requirements (summary)

| Category | Requirement |
|----------|-------------|
| Security | Secrets encrypted at rest; OAuth tokens never logged; least-privilege scopes. |
| Privacy | Personal data stays in Nikhil's own infra; no third-party data sharing beyond LLM API + chosen sources. |
| Reliability | Idempotent jobs; resumable workflows; no duplicate applications. |
| Observability | Structured logs, traces of agent decisions, metrics dashboards (Prometheus/Grafana). |
| Cost | Local mode ≈ infra-free; LLM spend capped via budget guard. |
| Performance | Recruiter-reply draft within 15 min of receipt; discovery cycle ≤ 30 min. |
| Maintainability | TypeScript, modular MCP tools, ADR-documented decisions. |
| Compliance | Respect each platform's ToS; honesty in all generated content. |
| Auditability | Every outward action has an immutable audit record + approval link. |

## 8. Constraints

- **C1 — ToS/legal:** Several job boards and LinkedIn prohibit automated scraping/applying. The system must prefer official APIs, partner feeds, RSS, and aggregators; browser automation is used conservatively and is configurable/disable-able per source. **This is a personal-use tool, not a mass-application or scraping service.**
- **C2 — Single user:** Designed for one operator (Nikhil); multi-tenant is out of scope.
- **C3 — Budget:** Must run for near-zero fixed infra cost locally.
- **C4 — Honesty:** Generated materials must never fabricate experience, skills, or credentials.
- **C5 — Human gate:** No outward action (apply/send/schedule) without an approval record in default mode.
- **C6 — Stack:** TypeScript/Node to match the maintainer's expertise.

## 9. Assumptions

- A1: Nikhil grants OAuth access to his own Gmail and Google Calendar.
- A2: A master profile/resume (MERN_Nikhil.pdf) is the source of truth for facts.
- A3: An Anthropic API key (Claude) is available for LLM calls.
- A4: At least one viable, ToS-compliant job source is configured.
- A5: Nikhil reviews the approval queue at least once daily.
- A6: Local machine or a small VPS is available to host Docker Compose.

## 10. Risks

| ID | Risk | Likelihood | Impact | Mitigation |
|----|------|-----------|--------|------------|
| R1 | Platform ToS violation / account ban (LinkedIn, boards) | Med | High | Prefer APIs/feeds; conservative rate limits; per-source toggles; HITL; clear personal-use scope |
| R2 | Embarrassing automated message (wrong thread/recipient) | Med | High | HITL approval on all sends; recipient/thread validation; dry-run mode |
| R3 | Resume/cover-letter fabrication damaging credibility | Med | High | Fact-grounded generation; profile as single source of truth; validation pass |
| R4 | LLM cost runaway | Med | Med | Budget guard, caching, model tiering, batching |
| R5 | Secret/token leakage | Low | High | Encrypted secrets, no token logging, scoped OAuth, audit |
| R6 | CAPTCHA / bot detection blocking automation | High | Med | Graceful degradation to manual hand-off; never auto-solve in violation of ToS |
| R7 | Duplicate / spammy applications | Med | Med | Idempotency keys, dedupe, per-company throttles |
| R8 | Data loss | Low | Med | Backups, migrations, point-in-time recovery |
| R9 | Over-automation eroding human judgment | Med | Med | HITL default; explainable scores; easy override |

## 11. Future Roadmap

- **Phase A (MVP):** Discovery + matching + tailoring + HITL apply + tracking (one or two sources).
- **Phase B:** Gmail monitoring + reply drafting + calendar scheduling.
- **Phase C:** Learning loop (outcome-driven matching/writing improvements) + analytics dashboard.
- **Phase D:** Semi-autonomous mode with confidence thresholds; more sources; A/B testing of resume variants.
- **Phase E (stretch):** Cloud deployment, mobile push approvals, interview-prep assistant, salary-negotiation assistant.

## 12. Out of Scope (v1)

- Multi-user / SaaS productization.
- Auto-solving CAPTCHAs or evading bot detection.
- Mass-applying / spray-and-pray.
- Acting on behalf of anyone other than Nikhil.
- Misrepresenting facts in any communication.
